# Маппинг координат между камерами холодильника

Решение задачи маппинга пиксельных координат из камер `top`/`bottom` на вид с камеры `door2`.

---

## Финальные метрики (val-сплит)

| Камера | MED | P50 | P90 | % < 50px | % < 100px |
|--------|-----|-----|-----|----------|-----------|
| top → door2 | **127.8 px** | 94.7 px | 268.0 px | 20.7% | 53.2% |
| bottom → door2 | **139.3 px** | 102.7 px | 284.0 px | 21.3% | 48.8% |

---

## Подход

### Модель: ZonalEnsemble

Задача — геометрический маппинг между камерами с разными углами и расстояниями до объектов. Глубина объектов в холодильнике вносит нелинейные перспективные искажения, которые нарастают к краям кадра (верх-право: 276 px, низ-центр: 58 px).

**Архитектура:**

```
ZonalEnsemble
├── GlobalSparseTPS(n_ctrl=300)     ← глобальный fallback
└── [Z зональных SparseTPS]        ← локальные модели
    ├── Зона 1: SparseTPS(n_ctrl=80) на точках этой зоны
    ├── Зона 2: SparseTPS(n_ctrl=80)
    └── ...

Предсказание = Σ w_z * pred_z,  где w_z = Gaussian(dist(x, center_z))
```

**Почему именно так:**
- Одна глобальная модель не справляется с нелинейностью в углах кадра
- TPS на всех точках численно нестабилен → SparseTPS с k-means опорными точками
- Зональное разбиение позволяет каждой зоне специализироваться на своей части пространства
- Gaussian-взвешивание обеспечивает плавные переходы на границах зон

**Что пробовалось:**

| Метод | CV MED (top) | CV MED (bottom) |
|-------|-------------|-----------------|
| Гомография (RANSAC) | 928 px | 668 px |
| Poly2 (без нормализации) | 224 px | 209 px |
| Poly2 (с нормализацией) | 213 px | 182 px |
| SparseTPS global | 178 px | 143 px |
| **ZonalEnsemble** | **177 px** | **142 px** |

### Анализ ошибок

Train-сессии дают MED ~120–160 px — такой же как val. Это означает **отсутствие переобучения**: модель упёрлась в физический потолок данных. Основная причина — межсессионный сдвиг камер (std по сессиям = 35–48 px) и нелинейность перспективы в угловых зонах кадра.

## Структура

```
solution/
├── train.py        # обучение, grid search, сохранение артефактов

├── evaluate.py     # MED на val/, сохраняет results/metrics.json

```

## Воспроизведение

### 1. Установка зависимостей

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

### 2. Обучение

```bash
cd solution
python train.py
```

Ожидаемое время: ~3–5 минут (grid search по 5 конфигурациям × 5 folds).

Артефакты сохраняются в `artifacts/`:
- `mapper_top.pkl`
- `mapper_bottom.pkl`
- `train_summary.json`

### 3. Оценка на валидации

```bash
python evaluate.py
```

Результат в `results/metrics.json`.

### 4. Анализ ошибок (опционально)

```bash
python analyze.py
```

## Использование predict()

```python
from predict import predict, predict_batch

# Одна точка
x_door2, y_door2 = predict(743.96, 524.59, source="top")
x_door2, y_door2 = predict(512.0,  300.0,  source="bottom")

# Батч точек
results = predict_batch([(743.96, 524.59), (920.42, 343.37)], source="top")
# → [(x1', y1'), (x2', y2')]
```

### Аргументы

| Параметр | Тип | Описание |
|----------|-----|----------|
| `x` | float | X-координата на кадре камеры source (пиксели) |
| `y` | float | Y-координата на кадре камеры source (пиксели) |
| `source` | str | `"top"` или `"bottom"` |

### Возвращает

`(x', y')` — предсказанная пиксельная координата на кадре `door2`

## Возможные улучшения

1. **Фичи изображений** — SIFT/SuperPoint дескрипторы для автоматического поиска соответствий между кадрами новой сессии и train → per-session адаптация без ручной разметки
2. **Нейросетевой регрессор** — при наличии 10× больше данных
3. **Калибровка камер** — если известны внутренние параметры камер, можно использовать эпиполярную геометрию
